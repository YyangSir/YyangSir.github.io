CAP 与 BASE 理论

发表于 2019-10-20

当系统由集中式转为分布式的过程当中会出现很多问题,其中必然会引入网络因素,在传统的请求中响应结果不是成功就是失败.然而因为网络的因素,响应结果将会多出超时这个状态.有时还会因为网络问题造成只有部分节点间可以通信俗称脑裂.或者一个节点直接宕机.在这样的条件下要实现一套严格满足 ACID 特性的事务时,就会出现系统的可用性和严格一致性之间出现冲突,也就是著名的 CAP 定理

CAP 定理

cap 理论告诉我们:一个分布式系统不可能同时满足一致性(C:Consistency),可用性(A:Availability),和分区容错性(P:Partition tolerance)这三个基本需求.

一致性

在分布式的环境中,一份数据在修改之后这份数据在不同节点的副本都可以保持一致.所有的用户可以从任意节点读取这份数据的最新值,这样的系统就具有强一致性.

可用性

系统提供的服务必须一致处于可用的状态,即:对于用户的每个操作都在有限的时间(用户期望)内返回结果.

分区容错性

分布式系统在遇到任何网络故障的时候,仍然需要对外提供满足一致性和可用性的服务.除非是整个网络出现故障.

网络分区是指在分布式系统中,不同节点的子网络,机房或者异地网络中由于一些原因造成子网络之前网络不连通的状况.但子网络内部是正常的.

对于分布式系统而言,网络问题是一个必然会出现的异常情况,分区容错性也就是必然要解决的问题.因此设计系统时,往往需要根据系统业务的特点,在一致性与可用性之间寻求平衡.

BASE理论

base 理论是对 cap 中一致性和可用性权衡的结果,其来源于对大规模系统分布式的实践总结.其含义为:基本可用(Basically Available),软状态(Soft state),最终一致性(Eventually consistent)三个短语缩写.

基本可用

在分布式系统中出现不可预知故障的时候,允许损失部分可用性,如响应时间的损失从 0.5s 增加到 2s,或者某个功能页面被引导到其他的降级页面中.

软状态

软状态也被称为弱状态,允许系统中的数据存在中间状态,并且这个状态不会影响系统整体可用,允许不同节点的数据副本的同步存在延迟.

最终一致性

数据有修改后,经过一段时间的同步,最终能够达到一个一直的状态,不需要系统实时保证数据的强一致性.

关于线程池ThreadPoolExecutor

发表于 2019-10-15

在 Executor 中最核心的类是 ThreadPoolExecutor,是线程池的实现类,下面我们将详细介绍有关这个类的构造参数和基本实现.

通用构造参数

  • corePoolSize 核心线程池的大小
  • maximumPoolSize 最大线程池的大小
  • keepAliveTime 线程存活时间
  • unit 存活时间的单位
  • workQueue暂时保存任务的工作队列
  • threadFactory 线程创建工厂
  • hander 拒绝策略

在ThreadPoolExecutor被创建的时候,在线程池里是没有线程的.

当我们向线程池中提交任务的时候首先会判断核心线程池(corePoolSize)中的任务是否都在执行,如果不是则创建新的线程执行任务.

如果核心线程池满了,则判断工作队列(workQueue)是否已满,没满则储存在工作队列中.

如果工作队列满了,则判断当前线程数是否等于最大线程池的大小(maximumPoolSize),如果小于则创建一个新的工作线程.否则就交给拒绝策略(hander)处理

这样的执行流程可以让线程池在执行 execute() 方法提交任务时尽可能地避免全局锁.

在线程池中的工作线程执行完任务后,经过一定的存活时间(keepAliveTime)后将会标记为可回收的.当然回收线程会产生额外的消耗.

几种默认线程池

  1. FixedThreadPool

    固定线程数的线程池,参数corePoolSize和maximumPoolSize相同,当线程池中线程满后,将后续任务加入队列.队列大小是 Integer.MAX_VALUE

  2. SingleThreadExecutor

    单个线程的线程池,队列大小为Integer.MAX_VALUE

  3. CachedThreadPool

    可缓存的线程池,corePoolSize设置为 0,maximumPoolSize为Integer.MAX_VALUE.队列使用的是没有容量的SynchronousQueue.keepAliveTime为 60,即:空闲线程超过 60s 将会被终止.

  4. ScheduledThreadPollExecutor

    固定长度的线程池,并且以定时或延迟的方式执行任务.

工作队列/阻塞队列

在 Java 包中提供了 7 个阻塞队列,分别如下:

  1. ArrayBlockingQueue

    一个有界数组组成的阻塞队列,默认线程以非公平的方式访问队列.即当队列可用时,阻塞的线程都可以争取访问队列的资格.

  2. LinkedBlockingQueue

    链表组成的有界阻塞队列,默认最大长度为 Integer.MAX_VALUES.

  3. PriorityBlockingQueue

    支持优先级的无界阻塞队列,可自定义 compareTo()方法来指定元素排序规则.无法保证同优先级元素顺序!

  4. DelayQueue

    支持延时获取元素的无界阻塞队列,并且可以自定义 compareTo()方法来指定元素顺序.加入此队列必须实现 Delayed接口.

  5. SynchronousQueue

    不存储元素的阻塞队列,每个 put 操作必须等待一个 take 操作,仅用来传递元素.默认采用非公平的访问策略.

  6. LinkedTransferQueue

    链表组成的无界阻塞队列.多出了特有的两个方法:tryTransfer()和 transfer()

  7. LinkedBlockingDeque

    链表组成的双向阻塞队列.

线程工厂

推荐使用 google 的guava提供的 ThreadFactoryBuilder 给线程设置名字.

拒绝策略

当线程池和队列都满了的情况下,再往线程池中添加任务则会执行下列策略:

  • AbortPolicy:直接抛出异常
  • CallerRunsPolicy:调用者所在线程执行任务
  • DiscardOldestPolicy:丢弃队列里最近的一个任务,并执行当前任务.
  • DiscardPolicy:丢弃任务

同时也可以实现RejectedExecutionHandler来自定义策略.

Redis 主从同步

发表于 2019-10-13

为了保证高可用,Redis 的主从同步是大多数公司都会做的方案,当 master 挂掉的时候让 slave 去接管,保证服务的继续.在 Redis 重启需要恢复数据时则需要将部分差异数据同步过来.下面将简单介绍这个过程.

增量同步

Reids 同步的指令流,主节点会将修改性的指令记录在本地的内存 buffer中,然后异步将 buffer 中的指令同步到从节点,从节点会记录一个偏移量,向主节点反馈自己同步到哪里了.

因为内存 buffer 是有限的,所以主节点不会把指令都记录在内存中.而是保留一个定长数组,如果内存 buffer满了就会从头开始覆盖前面的内容造成数据不一致

快照同步

快照同步非常消耗资源.他首先在主节点上进行 bgsave操作,将当前内存的数据全部保存到磁盘文件中,然后将磁盘文件传送给从节点.从节点将之前数据全部清空,然后加载快照中的数据.加载完毕后通知主节点继续进行增量同步.如果这时需要同步的指令在主节点中被覆盖了,那将会再次发起快照同步,陷入死循环.所以设置一个合适的 buffer 大小是很重要的.

当有新的节点加入到集群中,必须先进行一次快照同步,之后进行增量同步.

无盘复制

在 Redis 2.8.18 版本后,Redis 支持无盘复制.主节点直接通过套接字将快照内容发送给从节点.并且这个过程是一边遍历内存一边将序列化的内容发送到 从节点.

RabbitMQ 的可靠性传输

发表于 2019-09-23

首先分析下 RabbitMQ 丢数据的几种情况:

  1. 生产者丢数据
  2. RabbitMQ 丢数据
  3. 消费者丢数据

弄清楚丢数据的几种情况后我们逐个分析下

生产者丢数据

消息在发送给 RabbitMQ 的过程中数据因为各种原因丢失了,这时可以在生产者中开启confirm模式.每次写入消息RabbitMQ 都会异步回传一个 ack信息告诉生产者消息写入成功了.如果 RabbitMQ 没能处理消息则会回调 nack接口通知消息接收失败.

RabbitMQ 丢数据

RabbitMQ 自己丢数据则需要开启持久化,将消息写入磁盘,这样RabbitMQ 重启后会自己读取之前的数据.要注意的是需要同时开启 queue 元数据的持久化和 queue 中数据消息的持久化,才能保证数据不丢失.

消费者丢数据

这个时候就需要 RabbitMQ 的手动 ack机制了,必须关了 RabbitMQ 的自动 ack.在业务逻辑处理完成之后在程序里手动 ack通知 RabbitMQ 消息已处理.否则 RabbitMQ 会把这个消息分配给别的消费者处理.

Redis 的持久化

发表于 2019-08-14

Redis 通常作为缓存使用,当我们需要 Redis 有一定可靠性的时候,就需要对其进行高可用的升级.而数据的持久化是重要的一环.

RDB方案

Redis 提供的rdb方式可以将内存中的数据以二进制镜像形式保存在硬盘中.可以通过命令save触发,也可以通过配置文件定时触发.执行命令持久化时在数据保存完成之前会一直阻塞 redis.

在持久化的过程中 Redis内部会执行bgsave 命令去fork一个子进程(这个过程会阻塞 Redis),由子进程将数据写到rdb文件中.父进程和子进程共享数据段,父进程还可以继续对外提供读写服务.当数据有修改时父进程会将数据拷贝过来.而子进程的数据没有变化,这个过程被称为Copy On Write简称 COW.因此持久化的数据在子进程创建后就不会改变了,所以这种方式也被称为”快照”.显然这样会丢失部分数据.

AOF 方案

aof 方案相对于 rdb 有着更好的实时性.存储的是 Redis 服务器的顺序写指令.也就是记录增删改指令,读指令不会记录,在收到指令后,会先执行指令然后将指令保存到 aof 日志中.具体执行步骤如下:

  1. 服务器执行命令
  2. 将命令追加到 aof 的内存缓存区
  3. 调用 fsync 函数将缓冲区的函数写入到文件

其中第3 步的操作会很慢,因为涉及到了 IO 操作,所以并不是每条指令执行过后都会执行步骤 3.而是每秒调用 一次fsync函数将缓存刷到磁盘文件,这个时间可以通过配置文件设置.

随着 Redis 的运行aof日志将会越来越大,服务器将会对日志进行重写,以减小文件大小.具体执行步骤如下:

  1. 先fork一个子进程
  2. 遍历扫描 Redis 数据库
  3. 根据键值类型进行重写,序列化到一个新的aof日志文件中
  4. 序列化完毕后再将这个过程中发生增量的指令写入
  5. 使用新的日志文件替换原先的日志文件

混合持久化方案

很明显 aof 方案要比 rdb 好得多,因为rdb会丢失大量数据.但是相同数据量的情况下 aof文件会比 rdb大得多,数据恢复起来也要慢的多.为了解决这个问题,在 Redis 4.0之后出现一个一种混合持久化的方式.使用rdb全量备份数据,然后使用aof备份那些生成”快照”之后增加的数据.

在InnoDB执行普通查询操作读取被锁的那个记录时,为了不释放记录上的锁会生成一个版本快照(ReadView),读取操作就可以直接返回快照数据,而不需要等待锁的释放,提升了性能.由于并发操作一行记录不只有一个快照,由此带来的并发控制称为多版本并发控制即 MVCC(Multi-Version Concurrency Control).

此外,只有在 READ COMMITTD和 REPATABLE READ 两种隔离级别下会有.并且因为生产版本快照的时机不同这两种级别读取的数据也是不一样的.

READ COMMITTD

在每次进行查询操作之前都会生产一个版本快照,所以读取到的总是新的快照数据.

REPATABLE READ

在第一次查询操作前会生成一个版本快照,之后的查询会重复使用.所以读取到的是最开始的数据版本.

你确定不来了解下 Redis 跳跃表的原理吗

发表于 2019-05-26

前言

本章将介绍 Redis中 set 和 zset的基本使用和内部原理.因为这两种数据结构有很多相似的地方所以把他们放到一章中介绍.并且重点介绍zset 内部一个很重要的数据结构:跳跃表.

基本介绍

set

先来看看 set

Redis 中 set 集合很像Java 中 HashSet,键值对无序、唯一、不为空.

zset

zset 是 Redis 中最特别的基础数据结构,其他几个都能和 Java 大致对应上.它基本上还是一个 set 但是添加了一个 score 属性去保证有序性.其内部实现为跳跃表稍后将会着重介绍.

在 zset 中 score 的类型为 double 所以有时会出现小数点精度问题.

当 zset 中最后一个 value 被删除后,这个和 zset 就会被自动删除,内存被回收.

内部原理

Redis 的 zset 是个复合结构,是由一个 hash 和 skiplist 组成的,其中 hash 用来保存 value 和 score 对应关系.skiplist 用来给 score 排序.关于hash 的内部实现请参阅之前的一篇文章:《你确定不来了解一下Redis中 Hash的原理吗》,在这里我们着重介绍 skiplist 的实现.

skiplist 跳跃表

因为zset需要高效的插入和删除,所以底层不适合使用数组实现,需要使用链表的结构.当插入新元素时需要根据 score插入到链表合适的位置,保证链表的有序性.高效的办法是通过二分查找去找到插入点.

那么问题就来了,二分查找的对象必须是有序数组,只有数组支持快速定位,链表做不到该怎么办呢?这时,就该跳跃表出场了.

如图所示,跳跃表在链表的基础上加入了层级L0~L3的概念,Redis 的跳跃表共有 64 层,可容纳 $2^{64}$ 个元素.每个元素的层级是随机分配的,分配 L0 的概率是 100%,就是说每个元素至少会有一层.分配L1 的概率是 50%,分配 L2 的概率是 25%,往上以此类推.

每个 kv 对应的结构为zslnode.kv 之间使用指针形成有序的双向链表.同一层的 kv 会使用指针串起来.每层元素的遍历都是从跳跃表的头指针 kv header 出发.

header 的结构也是 zslnode,当中 value 为 null,score 为 Double.MIN_VALUE排在最前面.

查找

介绍完 skiplist的数据结构后,我们来具体看下skiplist 是怎样快速定位元素的.

在上图中,假设我们要查找 3 这个节点.skiplist 会从 header 的顶层出发遍历搜索找到第一个比目标元素小的开始降一层,直到降到最底层找到 3这个节点,搜索路径为:

  1. L3:header -> 4 -> header L3 层header找到 4比 3 大,回退到 header 同时降一层
  2. L2:header -> 2 -> 4 -> 2 L2层header 找到 2 比 3 小,继续遍历找到 4,回退到 2 同时降一层
  3. L1: 2 -> 4 ->2 L1 层 2 找到 4 比 3 大,回退到 2 降一层
  4. L0: 2 -> 3 L0 层 2 找到 3 期望节点

整个查找过程算法的时间复杂度为 $O(lg(n))$.

你确定不来了解一下Redis中 Hash的原理吗

发表于 2019-05-26

前言

本章接着上一节继续介绍 Redis 的基础数据结构中的Hash字典.

基本介绍

Hash 也可以用来存储用户信息,和 String 不同的是 Hash 可以对用户信息的每个字段单独存储,String 则需要序列化用户的所有字段后存储.并且 String 需要以整个字符串的形式获取用户,而 hash可以只获取部分数据,从而节约网络流量.不过 hash 内存占用要大于 String,这是 hash 的缺点.

Redis 中的 Hash和 Java的 HashMap 更加相似,都是数组+链表的结构.当发生 hash 碰撞时将会把元素追加到链表上.值得注意的是在 Redis 的 Hash 中 value 只能是字符串.

看完基本介绍之后,我们先来了解下 hash 的内部结构.第一维是数组,第二维是链表.组成一个 hashtable.

当hashtable 中元素逐渐变少时,Redis 会进行缩容来减少空间占用,并且缩容不会受 bgsave 的影响,缩容条件是元素个数少于数组长度的 10%.

你确定不来了解下 Redis 字符串的原理吗

发表于 2019-05-18

前言

Redis 有五种基础数据结构:string,list,set,zset,hash.其中 string是最最最简单的也是最常用的.这个数据类型虽然简单但是内部的结构设计却很是精致.

基本介绍

相比于 Java,在 Redis 中 string 是可以修改的,是动态字符串(Simple Dynamic String 简称 SDS)他的内部结构更像是一个 ArrayList,维护一个字节数组并预分配冗余空间以减少内存的频繁分配.当字符串的长度小于 1MB时,每次扩容都是加倍现有的空间,如果字符串长度超过 1MB 时,每次扩容时只会扩展 1MB 的空间.

ps:字符串长度为最大长度 512MB.

上面是字符串的基本操作 命令mset 和 mget 可以对多个字符串读写 节省网络开销

不仅如此redis 的字符串还可以用来储存整数(更不像Java 的字符串了),并且可以自增操作.字符串保存整数类型的的范围在 $-2^{64}$至$2^{64}-1$

如果保存的数大于这个取值范围就会变成普通字符类型 无法自增操作.这将由字符串编码格式决定.

字符串由多个字节组成,每个字节有 8bit.这样的数据结构还可以当做 bitmap 去使用.

编码格式

Redis 字符串编码格式有这么几种:int 编码、embstr编码和raw 编码 下面就详细介绍下这几种编码的区别.

在这之前先要说说RedisObject. Redis 的对象头,所有的 Redis 对象都有下面这个头部结构.

int 编码

当储存的值是64 位有符号整数类型的时候将会采用 int 编码,这时可以使用键值自增操作.Redis 在启动时会建立1w 个redisObject共享对象下文会讲到,值在[0,1000)之间.如果存入整数的值在[0,1000)中Redis将不会创建新的对象,而是直接指向共享对象,键值不额外占用空间.

使用 object encoding命令可以查看编码格式 使用 debug object命令可以查看更多信息

可以看到 foo 和 foo2 都在0x7f44b020aca0这里指向的是同一个对象

embstr 编码

当存储的字符串长度较短时(len<=44 字节),Redis将会采用 embstr 编码.embstr 即embedded string 嵌入式的字符串.将SDS结构体嵌入RedisObject对象中, 使用 malloc 方法一次分配内存地址是连续的.

思考

为什么字符串会有两种格式 embstr 和格式和 raw分界线是 44 个字节?

Redis 默认的内存分配器jemalloc分配内存大小的单位是$2^n$次方,为了容纳一个完整的 embstr 对象,最少会分配 32 字节的空间,再长些就是 64 字节,再之后就认为这是一个大字符串不适合用 embstr 存储,而改用 raw 编码了.
